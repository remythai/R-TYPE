12 /**
13 * @class Motion
14 * @brief System that simulates entity movement using physics principles.
15 *
16 * The Motion system implements a simplified physics engine that handles:
17 * 1. **Deceleration**: Velocity naturally dampens when acceleration is zero
18 * 2. **Acceleration**: Applies acceleration forces to velocity each frame
19 * 3. **Speed clamping**: Constrains velocity to maximum speed limits
20 * 4. **Position updates**: Translates entities based on current velocity
21 * 5. **Boundary clamping**: Keeps entities within screen boundaries
22 *
23 * This creates natural, responsive movement with friction-like behavior.
24 *
25 * @details
26 * **Physics Pipeline (per frame):**
27 * 1. **Deceleration Phase**: Reduces velocity by 75% when no external force is applied
28 * - Positive velocities approach zero from above
29 * - Negative velocities approach zero from below
30 * - This simulates air resistance/friction
31 *
32 * 2. **Acceleration Phase**: Adds acceleration values to velocity
33 * - Clamped to [-speedMax, +speedMax] range
34 * - Prevents unlimited velocity buildup
35 *
36 * 3. **Position Update Phase**: Translates entity position by velocity
37 * - Position clamped to screen bounds: [0, screenSize]
38 * - Uses Renderable component for screen dimensions
39 *
40 * **Deceleration Formula:**
41 * ```
42 * vel_new = vel - (vel * 0.75)
43 * = vel * 0.25
44 * ```
45 * This exponential decay creates smooth stopping behavior.
46 *
47 * @requires
48 * - Position: Current location to update
49 * - Velocity: Current movement speed with speedMax limit
50 * - Acceleration: Forces to apply
51 * - Renderable: Screen size for boundary constraints
52 *
53 * @performance
54 * - Time Complexity: O(n) where n is number of moving entities
55 * - Space Complexity: O(1) per entity
56 * - No allocations or iterations beyond entity processing
57 *
58 * @see Position
59 * @see Velocity
60 * @see Acceleration
61 * @see Renderable
62 */

82 /**
83 * @brief Updates all entity positions based on physics simulation.
84 *
85 * Processes each entity with motion components through a three-phase physics pipeline:
86 *
87 * **Phase 1: Deceleration (Natural Friction)**
88 * - Reduces velocity by 75% per frame (multiplies by 0.25)
89 * - Applies independently to X and Y axes
90 * - Preserves sign: positive velocities stay positive, negative stay negative
91 * - Creates smooth deceleration when no input is applied
92 *
93 * **Phase 2: Acceleration (Force Application)**
94 * - Adds acceleration components to velocity
95 * - Clamps result to [-speedMax, +speedMax] range
96 * - Prevents unlimited acceleration buildup
97 *
98 * **Phase 3: Position Update (Movement)**
99 * - Translates position by velocity amount
100 * - Clamps position to screen bounds using Renderable dimensions
101 * - Ensures entities never leave the visible play area
102 *
103 * @param registry Reference to the ECS Registry (unused but required by interface).
104 * @param dt Delta time since last update in seconds (unused in current implementation).
105 *
106 * @details
107 * **Deceleration Behavior:**
108 * - Velocity of +100 becomes +25 (reduction of 75)
109 * - Velocity of -100 becomes -25 (reduction of 75)
110 * - Velocity of +1 becomes +0.25 → eventually +0 (clamped)
111 *
112 * **Speed Limiting:**
113 * - Applied after acceleration step
114 * - Uses Velocity::speedMax parameter
115 * - Prevents physics simulation instability
116 *
117 * **Boundary Constraints:**
118 * - Screen position: [0, screenSizeX] and [0, screenSizeY]
119 * - Entities cannot move beyond screen edges
120 * - Uses Renderable component for screen resolution
121 *
122 * @attention
123 * - Deceleration applies EVERY frame, not just when no input
124 * - Multiple inputs in same frame accumulate additively
125 * - Position clamping prevents off-screen rendering
126 *
127 * @note
128 * - dt parameter is unused; update rate should be consistent
129 * - Physics is frame-based, not time-stepped
130 *
131 * @example
132 * ```cpp
133 * // Entity with vel=(10, 0), acc=(0, 0), speedMax=20
134 * // Frame 1: vel becomes (2.5, 0)
135 * // Frame 2: vel becomes (0.625, 0)
136 * // Frame 3: vel becomes (0, 0) → stops
137 * ```
138 *
139 * @example
140 * ```cpp
141 * // Player at (500, 500) with vel=(0, 0), acc=(5, 0), speedMax=10
142 * // Frame: vel becomes (5, 0), pos becomes (505, 500)
143 * // Next frame: vel becomes (1.25, 0), pos becomes (506.25, 500)
144 * ```
145 *
146 * @see Registry::each
147 * @see Position
148 * @see Velocity
149 * @see Acceleration
150 */
151 void onUpdate(Registry& registry, float dt) {
152 updateCount++;
153
154 registry.each<Position, Velocity, Acceleration, Renderable>([dt](auto e, Position& pos, Velocity& vel, Acceleration& acc, Renderable& render) {
155 // Phase 1: Deceleration (75% reduction)
156 vel.x = vel.x > 0 ? std::max(float(vel.x - (vel.x * 0.75)), float(0)) : std::min(float(vel.x - (vel.x * 0.75)), float(0));
157 vel.y = vel.y > 0 ? std::max(float(vel.y - (vel.y * 0.75)), float(0)) : std::min(float(vel.y - (vel.y * 0.75)), float(0));
158
159 // Phase 2: Acceleration (apply forces and clamp to speed limit)
160 vel.x = std::clamp(vel.x + acc.x, -vel.speedMax, vel.speedMax);
161 vel.y = std::clamp(vel.y + acc.y, -vel.speedMax, vel.speedMax);
162
163 // Phase 3: Position update (translate and constrain to screen bounds)
164 pos.pos.x = std::clamp(pos.pos.x + vel.x, float(0), render.screenSizeX);
165 pos.pos.y = std::clamp(pos.pos.y + vel.y, float(0), render.screenSizeY);
166 });
167 }
